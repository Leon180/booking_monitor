# D4.1 — Operator runbook for rolling out the KKTIX 票種 alignment

## TL;DR

D4.1 ships a coupled migration + binary change. Run them in this order:

1. **Apply migration `000014_kktix_ticket_type_alignment.up.sql`.** Renames `event_sections` → `event_ticket_types`, adds price/currency/sale-window columns, renames `orders.section_id` → `orders.ticket_type_id`, adds `orders.amount_cents` + `orders.currency`.
2. **Deploy the new binary.** Rebuild + restart `app`, `payment_worker`, `recon`, `saga_watchdog`. fx will not start cleanly against pre-migration schema.
3. **(Optional) Backfill or expire any in-flight `awaiting_payment` orders that pre-date the migration** — see [§ Legacy-row recovery](#legacy-row-recovery) below.
4. **Remove the deprecated env vars `BOOKING_DEFAULT_TICKET_PRICE_CENTS` and `BOOKING_DEFAULT_CURRENCY`** from `.env` / configmaps / systemd unit files. The new binary emits a Stderr WARN line at startup if either is still set; ignore is safe but the warning will appear on every restart until cleaned up.

The order is load-bearing. Reverse it (binary first, migration second) and every booking attempt during the gap will silently fail with a `column "ticket_type_id" of relation "orders" does not exist` Postgres error, the worker will revert Redis inventory, and the customer's HTTP 202 booking will silently disappear before the worker can persist it.

## Why migrate first?

The new binary expects schema state matching `000014`:

- `internal/infrastructure/persistence/postgres/order_row.go` has `orderColumns = "id, event_id, user_id, quantity, status, created_at, reserved_until, payment_intent_id, ticket_type_id, amount_cents, currency"` and INSERTs a row with all 11 columns.
- `internal/infrastructure/persistence/postgres/repositories.go::postgresTicketTypeRepository` queries `event_ticket_types` by name, not `event_sections`.

A pre-migration binary boot will:

- Pass `/livez` (no DB query) but fail every `POST /api/v1/book` and `POST /api/v1/events` against the wrong-shape table.
- Not surface in `/readyz` because that endpoint only checks connectivity (`SELECT 1`), not DDL state. There is no automated schema-version probe — operator discipline is the only guard. (TODO in a future PR: add a migration-state check to `/readyz` so the binary reports `503 schema_stale` instead of accepting traffic.)

## Legacy-row recovery

After the migration, any orders that were in `awaiting_payment` BEFORE the migration ran will have:

```sql
SELECT count(*) FROM orders
 WHERE status = 'awaiting_payment'
   AND amount_cents IS NULL;
```

These rows cannot complete `/pay` — the post-D4.1 `PaymentService.CreatePaymentIntent` defensive `HasPriceSnapshot()` guard rejects them with `ErrOrderMissingPriceSnapshot` (mapped to HTTP 409 `"order price data unavailable; contact support"`, logged at Error in the service for paging visibility).

Three recovery options, in order of preference:

### Option 1 (recommended): wait for D6 expiry sweeper

D6 (the Pattern A reservation expiry sweeper) flips orders past their `reserved_until` to `expired` and triggers saga compensation, which reverts Redis inventory. **D6 is not yet implemented** as of D4.1; this option is forward-looking.

For deployments BEFORE D6 ships, choose Option 2 or 3.

### Option 2: backfill `amount_cents` from the default ticket type

For each affected order, look up the event's default ticket type and snapshot its price onto the order:

```sql
UPDATE orders o
   SET amount_cents = tt.price_cents * o.quantity,
       currency     = tt.currency,
       ticket_type_id = tt.id
  FROM event_ticket_types tt
 WHERE o.event_id = tt.event_id
   AND o.status = 'awaiting_payment'
   AND o.amount_cents IS NULL
   AND tt.name = 'Default';   -- D4.1 default ticket_type label; D8 may have multiple
```

Verify the affected count BEFORE running:

```sql
SELECT count(*) FROM orders o
  JOIN event_ticket_types tt ON tt.event_id = o.event_id AND tt.name = 'Default'
 WHERE o.status = 'awaiting_payment'
   AND o.amount_cents IS NULL;
```

After the backfill, those orders can complete `/pay` normally.

### Option 3: force-expire the affected rows + revert Redis inventory

For deployments where the legacy rows are stale (the `reserved_until` window has elapsed) or the operator prefers not to backfill:

```sql
-- Snapshot affected rows for compensation
CREATE TEMP TABLE legacy_expired_orders AS
SELECT id, event_id, quantity FROM orders
 WHERE status = 'awaiting_payment'
   AND amount_cents IS NULL;

-- Mark expired (status flip will trigger saga compensation on next watchdog sweep)
UPDATE orders SET status = 'expired'
 WHERE id IN (SELECT id FROM legacy_expired_orders);
```

Then run the saga watchdog manually (or wait for the next scheduled sweep — default 60s) to revert Redis inventory:

```bash
docker-compose exec saga_watchdog /app/booking-cli saga-watchdog --once
```

The customer must rebook from scratch.

## Detection alert

A Prometheus alert worth wiring (deferred to a follow-up PR with the alert rule itself):

```yaml
- alert: D4_1_LegacyRowsStuckAwaitingPayment
  expr: |
    sum(orders_status_count{status="awaiting_payment", amount_cents_null="true"}) > 0
  for: 5m
  labels: { severity: page }
  annotations:
    summary: "Pre-D4.1 awaiting_payment rows detected with no price snapshot"
    runbook_url: "https://github.com/Leon180/booking_monitor/blob/main/docs/runbooks/d4.1_rollout.md#legacy-row-recovery"
```

This alert requires a `orders_status_count{status, amount_cents_null}` gauge that doesn't exist yet — flagged for the post-D4.1 observability follow-up.

## Rollback

Migration `000014` has a `.down.sql` that reverses every step. The down migration is round-tripped in the integration test suite (`TestMigrationsRoundTrip`).

If you need to roll back the binary AFTER the migration ran:

1. Stop traffic (drain `/api/v1/book` at the load balancer).
2. Revert the binary to the pre-D4.1 image. The pre-D4.1 binary will boot against the post-migration schema (the new columns are NULLable; the renamed columns aren't queried by name in the pre-D4.1 code) but the API surface returns to the pre-D4.1 contract.
3. Run `migrate down 1` against `000014.down.sql`. This deletes the price/currency columns from `orders`, deletes the new columns from `event_ticket_types`, renames the table back to `event_sections`, renames the column back to `section_id`, and recreates the original index.
4. Resume traffic.

**Caveat**: any `event_ticket_types` rows created during the D4.1-deployed window will be preserved through the rename-back (the data survives), but their `price_cents` / `currency` / sale-window data is dropped on the down migration. If those columns held production data you cared about, snapshot the table before running the down migration.
