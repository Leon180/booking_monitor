End-to-end verification against `docker-compose` after rebuilding `app`,

branch. All 10 D4.1 commits applied.

## Scenario 1 — Happy path: events → book → pay

```http
POST /api/v1/events
{ "name": "D4.1 Smoke Test", "total_tickets": 50, "price_cents": 2000, "currency": "usd" }
→ 201
{ "id": "...", "ticket_types": [{ "id": "TT", "price_cents": 2000, "currency": "usd", ... }] }

POST /api/v1/book
{ "user_id": 42, "ticket_type_id": "TT", "quantity": 2 }
→ 202
{ "order_id": "O", "status": "reserved", "links": { "pay": "...", "self": "..." } }

GET /api/v1/orders/O
→ 200
{ "status": "awaiting_payment", "reserved_until": "...", "quantity": 2, ... }

POST /api/v1/orders/O/pay
→ 200
{ "amount_cents": 4000, "currency": "usd", "payment_intent_id": "pi_...", ... }
```

`amount_cents = 4000 = priceCents × quantity = 2000 × 2` — proves the
price snapshot is read off the Order, not from a global config. The
multiplication happens once, in `BookingService.BookTicket`, against
the ticket_type lookup; `PaymentService.CreatePaymentIntent` reads
the frozen snapshot via `order.AmountCents()`.

## Scenario 2 — Error mapping (negative cases)

| Case                                      | Expected | Actual | Source of rejection                 |
|---|---|---|---|
| `price_cents: 0`                          | 400      | 400    | `binding:"required,min=1"` in DTO   |
| `currency: "USDD"` (4 letters)            | 400      | 400    | `binding:"len=3"` in DTO            |
| `event_id` (pre-D4.1 field name)          | 400      | 400    | `ticket_type_id` required, missing  |
| `ticket_type_id: <non-existent UUID>`     | 404      | 404    | `domain.ErrTicketTypeNotFound` → mapError |

Pre-D4.1 clients sending the old `event_id` field correctly reject
with 400 — the breaking change is loud, not silent.
